\subsection{术语说明}

LHCb 升级二期中的径迹按穿越的子探测器组合可分为以下几类（参见 Figure.~\ref{fig:overview}）：
\begin{itemize}
  \item \textbf{Velo track}：仅穿过 TV 的径迹，由 TV standalone tracking 重建。
  \item \textbf{Upstream track}：穿过 TV 和 UP 但不穿过磁铁的径迹，由 Upstream tracking 将 TV track 与 UP hits 匹配得到。
  \item \textbf{Long track}：贯穿 TV、(UP)、磁铁及 MT（MP/FT）的径迹，由 Long tracking 重建，是物理分析中最常用的径迹类型。
  \item \textbf{Downstream track}：穿过 UP 和 MT 但不穿过 TV 的径迹（如 $K_S^0$、$\Lambda$ 等长寿命粒子衰变产物），由 Downstream tracking 重建。
  \item \textbf{T track}：仅穿过 MT（MP/FT）的径迹，由 MP/FT standalone tracking 重建。
\end{itemize}

\begin{lstlisting}[style=python, label=lst:up-config, caption={UP 探测器配置代码（UPConfig）}]
# Moore/Hlt/RecoConf/python/RecoConf/run5_defaults.py L103-127
class UPConfig(_PixelConfig):
    _DERIVED_KEYS = {"Sx", "Sy"}
    def __init__(self):
        self.PitchX = 0.050 * mm
        self.PitchY = 0.150 * mm
        self.LayerKeep = (
            True,
            True,
            True,
            True,
        )
        self.Eff = 0.98

    @property
    def Sx(self):
        return self.PitchX / sqrt(12)

    @property
    def Sy(self):
        return self.PitchY / sqrt(12)

DEFAULTS_UP = UPConfig()
\end{lstlisting}

\subsection{Downstream Tracking}

Downstream tracking 的目标是将 UP track 与 MT（MP + FT）的 standalone track 进行匹配，重建出穿过磁铁但不经过 TV 的 downstream track。其核心流程为：
\begin{enumerate}
  \item 以 UP track 为种子，向后匹配 FT/MP 的 standalone track，得到 downstream track；
  \item 合并 FT 与 MP 的 downstream track 得到 MT downstream track；
  \item 通过 Kalman Filter 拟合后，用 Long track 去重（declone）得到最终的最佳 downstream track。
\end{enumerate}

\subsection{Long Tracking}

Long tracking 的目标是将 TV track 与下游 MT（MP + FT）的 track 连接起来，重建出贯穿全探测器的 long track。目前实现了三种策略：
\begin{enumerate}
  \item \textbf{Backward Matching}：以 TV track 为种子，向后匹配 downstream track（Downstream-FT / Downstream-MP），添加 UP hits 形成 long track；
  \item \textbf{Forward Tracking}：以 Velo-UP track 为种子，向前扩展 FT/MP hits 形成 long track；
  \item \textbf{MatchNN\_UP}：用神经网络匹配 TV track 与 FT/MP standalone track，并可选添加 UP hits。注意，在默认的配置中，MatchNN\_UP 仅用于匹配 TV track 与 FT standalone track，不添加 UP hits。
\end{enumerate}

最后将三种来源的 long track 进行合并（merge）得到最终的 long track 集合。

\subsection{Kalman Filter}

在 Tracking Sequence 中，所有重建出的 track 都会经过 Kalman Filter 进行拟合（fit），以优化径迹参数。其调用链如下：

\textbf{第一步：全局拟合入口} —— \lstinline{get_fitted_all_tracks()} 遍历所有 track segment（Standalone、VeloUP、Long、Downstream），对每种 track 调用拟合。

\textbf{第二步：单类 track 拟合} —— \lstinline{get_fitted_and_selected_tracks()} 根据 track 类型（Velo-only 或 full track）选择不同的拟合路径：
\begin{itemize}
  \item \textbf{Velo-only}：调用 \lstinline{fit_tracks_velo_only()}，仅使用 TV hits 进行拟合；
  \item \textbf{Full track}：调用 \lstinline{fit_tracks()}，使用 TV + UP + MP/FT 的全部 hits 进行拟合。
\end{itemize}

\textbf{第三步：Fitter 配置} —— \lstinline{get_track_master_fitter()} 根据使用的子探测器（TV/UP/MP/FT）配置对应的 MeasurementProvider 和 MaterialLocator，返回配置好的 \texttt{TrackMasterFitter} 实例。

\textbf{第四步：执行拟合} —— \texttt{TrackEventFitter} / \texttt{TrackEventFitterFuture} 接收 tracks 和 fitter，完成 Kalman Filter 拟合。

\textbf{第五步：Chi2 筛选} —— 拟合后通过 \lstinline{select_tracks_chi2()} 调用 \texttt{TrackBestTrackCreator}，按 $\chi^2/\mathrm{DoF}$ 阈值筛选最佳 track。

\subsection{MC checking}

MC checking 的目标是利用 MC truth 信息对所有重建出的 track 进行效率（efficiency）、分辨率（resolution）等性能检查。在 Tracking Sequence 中，\lstinline{full_tracking_sequence()} 会对每种 track 调用 \lstinline{moni_efficiency()} 来配置 monitoring algorithm。

其调用链如下：

\textbf{第一步：MC 类别判定} —— \lstinline{get_moni_mc_hit_type()} 根据 track\_type / algorithm / sub\_det 返回 MC 类别标签（如 "Long"、"Downstream"、"Upstream"、"Velo"、"Seed" 等）及对应的子探测器列表。

\textbf{第二步：配置 efficiency checker} —— \lstinline{moni_efficiency()} 调用 \lstinline{check_tracking_efficiency()}，根据 track 容器类型选择 \texttt{PrTrackChecker}（v1::Track）或 \texttt{PrTrackChecker\_future}（future::Track）。

\textbf{第三步：MC 分类} —— \lstinline{get_mc_categories()} 根据 MC particle 的运动学信息将 track 分类（如 isLong、isDownstream 等），用于效率计算时的分类统计。

\textbf{第四步（可选）：分辨率检查} —— \lstinline{check_track_resolution()} 配置 \texttt{TrackResChecker} / \texttt{TrackResCheckerNT}，对 track 的 hit residual 和 momentum resolution 进行检查。

\section{Developer Guide}

\subsection{如何编译一个stack}

请先确保有如下环境之一：
\begin{itemize}
  \item CERN lxplus 或IHEP lxlogin()【请注意不要在EOS盘上编译，否则有概率碰到奇怪的问题，优先用online的机器或者afs】
  \item 自行安装双系统或者虚拟机，使用cern提供的almalinux9系统(推荐)；
  \item wsl + docker (作者没试过，但听说目前可行)。
\end{itemize}

本文的安装根据 \link{https://gitlab.cern.ch/lhcb-core/dev-tools/lb-stack-setup}{lb-stack-setup}，在 CERN lxplus 上编译一个 stack 的步骤如下：

新建一个目录

\begin{lstlisting}[language=bash]
  cd {workspace/}
  curl -sSL https://gitlab.cern.ch/lhcb-core/dev-tools/lb-stack-setup/raw/master/setup.py | python3 - stack
  cd stack
  vim utils/config.json
  make Moore # 或者你需要的其他软件包，例如Gauss, Boole, Detector等, 会自动编译依赖的软件包
\end{lstlisting}


然后会让你选择一个tag，默认为 \lstinline{lhcb-master}, 目前，常用的 tag 如下：

\begin{itemize}
  \item \lstinline{lhcb-run5}: 为二期升级所设置的分支，主要涵盖了 Moore 和 Rec 下的径迹重建算法，
  \item \lstinline{lhcb-sim11}: 主要包括了Simulation部分的软件，包括Gauss， Boole等。
  \item 目前有一个暂时使用的整合分支 \lstinline{lhcb-sim11-run5-gluing}，为以上两个分支的整合，主要用于测试和验证。更新会一点点落后于 \lstinline{lhcb-run5} 和 \lstinline{lhcb-sim11}。
\end{itemize}

对这些nightly的tag，有时候会出现无法编译，或者一些奇奇怪怪的情况，可以查看 \url{https://lhcb-nightlies.web.cern.ch/nightly/{tagname}}，看看对应的软件编译是否也有问题，若有可以在mattermost里面询问，有可能有修复的branch.

编译完成后，如果你想使用vscode 对于c++code 的智能提示，可以在workspace下创建一个 \lstinline{.vscode} 文件夹，并在里面创建一个 \lstinline{c_cpp_properties.json} 文件，内容类似如下（将[StackPath] 替换为你的实际路径， build 后面的编译版本用\lstinline{utils/config.json}决定）：

\begin{lstlisting}[language=json]
{
    "configurations": [
        {
            "name": "Linux",
            "includePath": [
                "${workspaceFolder}/**"
            ],
            "compileCommands": [
                "[StackPath]/Rec/build.x86_64_v3-el9-gcc15-opt/compile_commands.json",
                "[StackPath]/LHCb/build.x86_64_v3-el9-gcc15-opt/compile_commands.json",
                "[StackPath]/Detector/build.x86_64_v3-el9-gcc15-opt/compile_commands.json",
                "[StackPath]/Allen/build.x86_64_v3-el9-gcc15-opt/compile_commands.json",
                "[StackPath]/Lbcom/build.x86_64_v3-el9-gcc15-opt/compile_commands.json"
            ],
            "defines": [],
            "compilerPath": "/usr/bin/gcc",
            "cStandard": "c17",
            "cppStandard": "gnu++17",
            "intelliSenseMode": "linux-gcc-x64"
        }
    ],
    "version": 4
}
\end{lstlisting}

这样就可以在vscode中，就不会出现奇奇怪怪的红色报错波浪线了，也方便自动跳转寻找对应的变量定义.





\subsection{如何添加一个新算法}
完整的 tracking sequence 见 OptionFile \url{Moore/Hlt/Moore/run5/options/run5_all_tracking.py}，可帮助你了解整体流程。添加新算法的步骤如下：

\begin{enumerate}
	\item 在 \url{Rec/Pr/PrAlgorithms/src/} 下新建算法 cpp 文件，声明算法名称，并添加到 CMakeLists.txt 中确保编译。然后执行 \lstinline{make fast/Rec} 重新编译 Rec，使 Moore 可以调用该算法。
	\item 在 \url{Moore/Hlt/RecoConf/python/RecoConf/run5_default_tracking.py}（或其他 option 文件中，不要让一个文件管理所有配置——同一类配置放在一个文件，明显不一致时应讨论新建文件）中添加对应的配置函数，为新算法提供配置参数。
	\item 在你的 OptionFile 中使用（可参考 \url{run5_all_tracking.py} 的运行方式），确保你的函数能在 \lstinline{Reconstruction()} 中被执行。
\end{enumerate}


\subsection{如何打开一个MR，并请求合并自己的算法}


首先，将你写完的算法切换到一个新的分支，并提交到远程仓库：
\begin{lstlisting}[language=bash]
  git checkout -b {your_branch_name}
  pre-commit install # 安装 pre-commit 钩子，确保代码通过语法检查
  git add {your_new_algorithm.cpp}
  git commit -m "Add {your_new_algorithm}"
  git push origin {your_branch_name}
\end{lstlisting}


在 GitLab 上打开一个 MR（Merge Request），并在 MR 描述中说明你的算法的功能、用途以及任何需要注意的事项。确保你的 MR 目标分支是 \lstinline{lhcb-run5} 或其他相关分支。\@ 相关的reviewer 来审查你的代码。

上述操作皆可逆，无需过于担心，创建即可，会有人帮你改的. 即使 为了测试也可以打开一个MR，可以用 not-for-merge 的标签，或者在MR描述中说明， 同时不要求reviewer。
